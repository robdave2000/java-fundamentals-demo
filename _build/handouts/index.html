Intro

This project is tied to the course entitled “Fundamentals of Object Oriented Programming in Java” on Pluralsight. You can view the course here:

https://app.pluralsight.com/library/courses/fundamentals-object-oriented-programming-java/table-of-contents

Create a new Github repository called java-fundamentals-demo and get set up to be able to commit code locally on your machine to the repository.

Encapsulation

We have our droids, but they aren’t very useful they will need tools in order to do their jobs well. Lets create some for our droids to use, we’ll create a Tool class that has a two variables, a String toolType to represent what kind of tool it is, and an int durability to represent how many more operations the tool can handle before breaking.

A String type for toolType is fine, but there’s going to be many of these things, perhaps instead of a String, which we would have to define over and over again, perhaps we should only define it once. Create a Java Enum called ToolType and Add one entry for now called STARSHIP_REPAIR.

Don’t forget to add getters and setters to the tools class as well as a constructor. However, instead of allowing the user to specify the durability of the tool in the constructor, the constructor should default the durability to 3.

Lets add a list of tools to our droid class to represent the tools available to the droid. Add a new variable to the Droid class called “tools” and make it a list type. Remember to create getters and setters for it as well as adding to the constructor.

Great! we now need to change our Main class to be able to build our droid, now with tools. I’m continuing to build R2D2 so, as he is an astromech droid, I’ll give him 3 Starship repair tools so he can do 9 operations before he’s out of tools.

R2D2 needs a way to use his repair tools. Lets create a method in the Droid class called repairStarship, it should take the first available tool that has a STARSHIP_REPAIR as it’s ToolType and decrease the durability by 1, it should then print a message that the starship is repaired. If the durability reaches the tool should be removed. If no tool is available, the user should be informed. Try to make this method yourself and reference the solution code if you need some help.

Defending Encapsulation

We’ve created R2D2 and we’ve had him do some work. If you’re following along with the exercise files we have some work to do to defend our encapsulation. We need to be able to be certain that objects can only modify their own values, and collections should be using defensive copying. Also, anything that can be made immutable (unable to be changed after being set) should be made so.

Lets modify each of our classes to incorporate these rules.

Tool: the type of the tool shouldn’t change after being made, lets remove the setter and make the field final. Also anytime this tool is used, it should control its durability loss, lets remove the setDurability method and instead create a useTool method that will lower the durability for us. We’ll need to change the code in the Droid’s repairStarship method as well.

Droid: A droid’s serial number shouldn’t change, lets make it final and remove it’s setters. A Droid should be creating a copy of the list of tools used to create it and make that list final so that a user cannot modify the list after it has been created. A droid’s model shouldn’t change after creation either, we’ll make it final and remove any setters. This by itself doesn’t completely help us unless we also make the fields on the Model final. This is because a user with access to the Model object itself could simply change the fields on the model object and that would be reflected in R2D2’s model. Setting the Droid’s model to final makes sure we can’t assign any other model, but we need to make sure that model doesn’t allow changes either. Lets do that now.

Model: A models type and purpose shouldn’t be changed after the model is created, lets make both fields final and remove the setters.

Abstract Classes and Inheritance

Up until now we’ve only worked with R2D2 even though we created 2 other types of droid, Protocol and Battle droids. We should allow our factory to build those droids as well. The problem is that our Droid class has code that is specific to the Astromech droid. We should probably have a specific class for each of the droids.

We also don’t want to repeat ourselves and each of our Droids needs to have their own serial number and should probably have the speak method to default to beeps and boops as well. We want protocol droids and battle droids to be able to use tools, but should speak in common languages as well. This means we should keep Droid as an abstract class. An abstract class cannot be created, but serves as a base for other classes that will be. Lets create 3 new Classes, AstromechDroid, ProtocolDroid, and BattleDroid and have them inherit from Droid.

This means that we need to change R2D2 in the main class to be of type AstromechDroid instead of Droid because abstract classes can no longer be instantiated, that is, created.

Protocol Droids don’t actually need tools as their main function is speaking and offering advice, but BattleDroids need to be able to shoot things and Astromech Droids need to repair things. Lets move reepairStarship to the AstromechDroid class. In order to do this we have to change Droid’s tools variable to be protected instead of private. This simply allows us to access tools from the super class, Droid, in the subclass, AstromechDroid.

Lets create a new set of tools for our battle droids, we’ll call them rifles and we’ll create a new method on BattleDroid called shootEnemy. It does basically the same thing as the repairStarship method, but we’ll print that an enemy has been shot instead of a starship repaired.

Lets add 2 new droids to our main class. We’ll create C3PO for our protocol Droid, and OOM-9 as our battle droid.

Lets make a list of Droid types by referencing the Droid superclass and add each of our specialized droids to the list before we have them do any special work like repairing starships or shooting enemies. Now we’ll see if they can each remember how to speak as they should! Remove all previous references to the speak method and instead create a for loop that loops over each droid and calls their speak method.

We can see that Droids are still Droids even if they have specialized code. We also see that the code Overrides still fuction even if we use them from the reference point of a superclass.

Interfaces

We’re almost done, we just need to talk about interfaces. Interfaces let us assign common functionality to objects that may not be so similar. Lets create an interface called Defender that will have a singular method, defend.

The defend method will implement a default method (only available as of java 8). This assumes that the only available way for a droid to defend itself is to shock it’s enemy. We’ll print out to the console that the droid shocked the enemy as the default.

2 of our droids can defend themselves, the BattleDroid and the AstromechDroid. Lets have both of them implement Defendable. Our ProtocolDroid is too fearful. AstromechDroid’s only have access to a shcck attach so implementing the interface is enough. The Default method will take care of it’s functionality. The BattleDroid has a rifle thoguh! Lets have him use that. We can override defend in BattleDroid and simply have it call it’s shootEnemy method. Simple.

Lets have them show off their abilities in the Main Method, Lets create 2 more Droids, an Astromech, R2D5, and a Battle Droid, OOM-7. Assign them both to a new List of Defenders (note that C3P0 can’t be added to a List<Defender>, ProtocolDroids don’t implment the interface.) and then loop over them having them call their new defend method.
